# 演示路线

## 文档目的

目标：

- 说明公开演示时应该如何从页面能力一路讲到库存、订单、支付和异步链路

操作步骤：

1. 先确认当前是 Mock 聚合模式还是真实微服务模式
2. 再按本文给出的演示顺序逐段展开
3. 对每一段都保留页面证据和接口证据

预期结果：

- 读者能够理解项目不是单纯页面展示，而是完整交易链路样例

可观察证据：

- README 截图
- `docs/` 文档
- 真实服务日志与接口返回

## 演示前准备

目标：

- 保证演示前环境一致，避免截图、订单状态和库存状态互相污染

操作步骤：

1. 准备前端开发服务器
2. 准备 Mock Service 或真实微服务链路
3. 如果演示真实异步下单，确认 Redis、RabbitMQ、MySQL 可用
4. 如需重置 Mock 演示态，清理运行时 `data/` 目录中的临时 JSON

预期结果：

- 首页、分类页、购物车、结算页和订单页都能稳定打开

可观察证据：

- 前端页面可访问
- Mock Service 或真实服务端口正常监听

## 演示 4：购物车库存校验

可观察证据：

- 页面提示信息
- 提交按钮阻断结果

## 演示 8：异步下单与订单状态轮询

目标：

- 展示异步下单的完整处理链路，包括前端轮询状态页

操作步骤：

1. 调用 `GET /member/order/token`
2. 调用 `POST /member/order/async`
3. 观察返回 `PROCESSING`
4. 轮询 `GET /member/order/process/{orderNo}`
5. 成功后进入支付页或订单详情页

预期结果：

- API 先返回处理中
- MQ 消费后状态变为成功
- 前端能够感知 `PROCESSING / SUCCESS / FAILED`

可观察证据：

- `ORDER2026062089652842`
- `ORDER2026062094999087`
- 本地 `order-service` 日志中的 producer / consumer 记录

## 演示 9：库存取消回补与支付确认扣减

目标：

- 展示库存状态会随着取消订单和确认支付发生流转，而不是单向扣减

操作步骤：

1. 创建一笔待支付订单
2. 调用 `PUT /member/order/{orderNo}/cancel`
3. 再创建一笔订单并完成支付
4. 对比 Redis 和数据库库存字段变化

预期结果：

- 取消后库存回补
- 支付成功后锁定库存转为已售库存

可观察证据：

- `ORDER2026062054143087`
- `ORDER2026062042352606`
- Redis 库存与数据库库存字段变化

## 演示 10：低库存并发压测

目标：

- 演示在低库存并发场景下不会超卖

操作步骤：

1. 将 `skuId=1027026` 收敛到受控基线 `20/10/1/9`
2. 以 `20` 个并发请求调用异步下单
3. 统计成功和失败数量
4. 取消成功订单并恢复库存

预期结果：

- 只有 `10` 笔成功
- Redis 不会出现负数
- 数据库不会出现负数
- 最终库存可恢复

可观察证据：

- 成功 `10`、失败 `10`
- Redis：`10 -> 0 -> 10`
- DB：`20/10/1/9 -> 20/0/11/9 -> 20/10/1/9`
- 代表订单：
  - `ORDER2026062030330517`
  - `ORDER2026062064348715`
  - `ORDER2026062042706069`

## 常见问题

目标：

- 提前说明最容易影响演示稳定性的环境问题

操作步骤：

1. 检查 Redis、RabbitMQ、MySQL 是否已启动
2. 检查前端代理和 Mock Service 端口是否一致
3. 检查运行时订单和权益数据是否污染当前演示

预期结果：

- 演示前能快速识别环境问题，不把环境问题误判为功能缺陷

可观察证据：

- 服务端口监听状态
- 运行时数据文件状态
