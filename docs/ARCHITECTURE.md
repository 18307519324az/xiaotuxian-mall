# 架构说明

## 文档目的

本文用于说明主仓库为什么被定义为“完整平台主仓”，以及真实微服务链路、本地 Mock Service 和 PC 商城前端在公开展示中的关系。它不是部署手册，也不是某个单独服务的开发文档，而是面向公开仓库读者的整体视图说明。

## 仓库定位

`xiaotuxian-mall` 不是单纯的“后端仓库”，也不是只包含某一个服务的最小示例。它承担的是完整平台主仓的职责，统一承载：

- 真实微服务后端链路
- 本地 Mock Service
- 前端能力说明与演示入口
- SQL 初始化脚本
- 演示、验收、面试与发布说明文档

这种组织方式的目标不是最小化代码体积，而是让公开访问者能够在一个仓库里看清平台能力边界和各部分角色。

## 三类能力分层

主仓需要同时说明三层能力：

1. 真实微服务后端
2. 本地 Mock Service
3. PC 商城前端

如果把三者混成一套后端技术栈，会让读者误解 Mock Service 也依赖 MySQL、Redis、RabbitMQ，或者误解前端仓只是附带页面。公开 README 必须把这三层拆开写清楚。

## 真实微服务后端

真实微服务链路负责交易核心能力，重点包括：

- 商品、分类、购物车、库存、订单、支付、会员等服务拆分
- Nacos 服务注册发现
- OpenFeign 服务间调用
- Redis 缓存与库存预扣
- RabbitMQ 异步下单
- MySQL 落库与库存状态流转
- OpenAPI / Knife4j 接口说明

这部分链路用于说明真实服务模式下的库存、订单、支付与异步处理能力，也是主仓技术含量最高的部分。

## 本地 Mock Service

Mock Service 的职责不是模拟全部基础设施，而是作为独立可运行的本地接口服务，为前端提供低成本演示环境。它的特点是：

- 基于 Spring Boot 独立启动
- 使用 JSON 文件作为基础数据源
- 不依赖 MySQL、Redis、RabbitMQ
- 不承担真实微服务的注册发现、消息消费和库存预扣逻辑

它的存在是为了提高联调效率和公开演示稳定性，而不是替代真实后端。

## PC 商城前端

前端层负责读者最直观看到的用户流程，包括：

- 首页、分类、详情、购物车、结算、支付、订单详情、会员中心
- 商品列表库存排序与仅看有货
- 商品详情库存展示与低库存提示
- 购物车与结算页库存校验
- 优惠券、礼品卡、货到付款手续费联动
- 同步下单与异步下单请求封装
- 异步下单状态轮询

前端并不直接承担库存扣减与消息处理，但它承担“页面如何感知并消费这些能力”的展示职责。

## 双模式运行的必要性

项目保留 Mock 模式和真实服务模式，不是为了堆技术，而是为了解决两个公开展示场景：

- 快速演示：直接起 Mock Service 和前端，页面流程稳定、启动成本低
- 深度讲解：切换到真实微服务后端，展示 Redis Lua、RabbitMQ、库存状态流转和订单幂等

这种双模式设计让仓库既能用于演示，也能用于面试说明。

## 典型调用链路

### Mock 模式

```text
浏览器页面
  -> 前端 API 封装
  -> 本地代理 /api
  -> Mock Service
  -> JSON 数据源
```

### 真实服务模式

```text
浏览器页面
  -> 前端 API 封装
  -> 真实微服务接口
  -> 订单服务 / 库存服务 / 支付服务 / 会员服务
  -> Redis / RabbitMQ / MySQL
```

前者强调“可演示”，后者强调“可解释真实实现”。

## 库存与订单链路在架构中的位置

主仓的核心竞争力不是单个页面，而是库存与订单链路的完整性：

- 下单前 Redis Lua 原子预扣
- 数据库库存预占
- RabbitMQ 异步创建订单
- 支付成功将 `lockedStock` 转为 `soldStock`
- 取消订单回补数据库与 Redis

因此主仓文档必须围绕库存、订单、权益结算来组织，而不是只列一串模块目录。

## 为什么要保留大量中文说明文档

公开仓库的主要阅读者通常不是直接进入源码的开发者，而是：

- 面试官
- 招聘方技术负责人
- 需要快速理解项目价值的外部读者

这些读者更需要一套分层清楚、文档成体系、中文可直接阅读的仓库说明，而不是只有代码目录。

## README 与独立 docs 的分工

README 负责快速建立认知：

- 仓库定位
- 三层技术栈
- 核心能力
- 截图与文档入口

独立 docs 负责展开说明：

- 演示路线
- 架构说明
- 库存与订单流程
- 验收清单
- 权益闭环
- 面试问答

这种分工可以避免 README 过长，也避免 docs 过碎。

## 公开展示目标

本仓库最终公开展示目标是：

- 让读者一眼看懂主仓、Mock、前端分别是什么
- 让技术栈描述不产生误解
- 让截图、文档、README、About、topics 相互一致
- 让平台能力能够用于简历、面试和对外演示

这也是本轮公开仓库整顿的最终落点。
