# 面试问答：异步下单与库存

## 异步下单链路

1. 前端获取下单 token
2. 后端校验商品、地址、权益参数
3. Redis Lua 原子预扣库存
4. 写入订单处理状态 `PROCESSING`
5. 发送 MQ 创建订单消息
6. 消费端创建真实订单并预占数据库库存
7. 前端轮询状态直到 `SUCCESS` 或 `FAILED`

## 为什么要这样拆

- 接口返回更快
- 高并发下入口压力更小
- 订单创建失败时有独立的回滚与补偿位置

## 库存状态模型

- `availableStock`
- `lockedStock`
- `soldStock`

状态流转：

- 下单成功：`available -> locked`
- 支付成功：`locked -> sold`
- 取消订单：`locked -> available`

## 受控并发验证结果

- SKU：`1027026`
- 受控基线：`20/10/1/9`
- 并发请求：`20`
- 成功：`10`
- 失败：`10`
- Redis：`10 -> 0 -> 10`
- DB：`20/10/1/9 -> 20/0/11/9 -> 20/10/1/9`

结论：

- 没有超卖
- 没有出现负库存
- 成功数没有超过初始可售库存

## 面试时怎么解释这组数据

这组数据最重要的不是“压了多少请求”，而是它能证明三件事：

- Redis 入口没有放大成功数
- 数据库状态流转保持守恒
- 取消或回滚后库存可以恢复

因此在面试中，重点应放在“为什么这样设计能避免超卖”，而不是单纯强调并发数字。

## 与同步下单的差异

同步下单更适合解释稳定兜底链路，异步下单更适合解释：

- 快速返回
- MQ 解耦
- 前端轮询
- 失败补偿

这也是项目保留两条下单路径的原因。
