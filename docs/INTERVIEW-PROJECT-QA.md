# 面试项目问答

## 项目整体架构是什么？

项目分为前端、Mock 聚合服务和真实微服务三层。前端负责页面体验与轮询流程，Mock 聚合服务负责完整演示链路，真实微服务负责库存、订单、支付、会员等关键能力。

## 为什么同时保留 Mock 模式和真实微服务模式？

Mock 模式更适合快速演示完整流程，真实微服务模式更适合讲解数据库、Redis、RabbitMQ 在交易链路中的作用。两者并存，可以兼顾演示效率和架构说明。

## 库存扣减怎么防超卖？

先用 Redis Lua 做原子预扣，快速挡住高并发竞争；再由数据库做预占与状态流转，确保最终一致性和可恢复性。受控并发测试里，成功订单数没有超过初始可售库存。

## 为什么 Redis Lua 之后还要数据库预占？

Redis 解决的是高并发入口争抢问题，数据库解决的是持久化状态、支付确认、取消回补和审计问题。只靠 Redis 无法完成完整订单生命周期管理。

## RabbitMQ 在异步下单中解决了什么问题？

它把“前端提交请求”和“真实订单创建”解耦，降低同步阻塞时间，让接口先快速返回 `PROCESSING`，再由消费端完成订单落库和库存预占。

## MQ 消费失败怎么办？

处理状态会标记为失败，并触发 Redis 预扣回滚；同时保留补偿处理路径，避免库存长期不一致。

## 重复提交怎么防？

前端每次下单都先获取一次性 token，后端消费 token 后立即失效，重复提交同一个 token 会被拦截。

## 重复消费怎么防？

异步链路使用 `requestId`、`orderNo` 唯一约束以及预占记录唯一约束，消费端会在重复请求下保持幂等。

## 取消订单如何回补库存？

待支付订单取消后，数据库会释放预占库存，Redis 也会按预扣数量回补。

## 支付成功为什么不回补 Redis？

支付成功说明库存已经从“占用”变成“售出”，这不是异常回滚场景，所以 Redis 不应恢复可售库存。

## 优惠券和礼品卡怎么参与同步与异步下单？

两条链路都会先完成金额计算，再把权益快照字段落到订单表中，支付页与订单详情页会回显这些字段。

## 零元支付怎么处理？

当礼品卡足以覆盖应付金额时，订单会直接进入已支付后的状态流转，不再依赖额外支付动作。

## 这个项目是不是完整秒杀系统？

不是。它不是完整秒杀活动系统，但使用小库存高并发下单场景验证了库存扣减链路：同一 SKU、小库存、高并发请求下，Redis 和数据库都没有出现负数，成功订单数没有超过初始可售库存。

## 还能继续优化什么？

- 增加更完整的死信重试观测
- 增加更细的链路监控和告警
- 提升发布脚本自动化程度
- 将更多会员能力从 Mock 切到真实服务

## 为什么这个项目适合做公开展示？

因为它同时具备三层可讲内容：

- 页面层：库存展示、结算权益、异步轮询
- 业务层：库存状态模型、订单状态流转、支付确认
- 工程层：Mock 模式、真实微服务模式、公开文档与演示路线

公开仓库如果只保留代码，很难让面试官迅速看清这些层次；而当前版本已经把这些结构补到了 README 和 docs 中。

## 为什么要保留公开中文文档？

真实面试和作品集场景里，很多读者不会先读源码，而是先看：

- README
- 架构说明
- 演示路线
- 验收报告

中文文档能直接降低理解门槛，也更适合解释项目边界、真实后端链路和 Mock 模式的关系。

## 如何用一句话概括这个项目？

可以概括为：

“一个围绕电商交易主链路构建的公开展示项目，前端负责库存和权益体验，后端负责 Redis Lua 防超卖、RabbitMQ 异步下单以及订单状态流转，Mock Service 用于稳定联调和页面演示。”
